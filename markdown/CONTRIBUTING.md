This project and everyone participating in it is governed by our [Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## How Can I Contribute?

### Reporting Bugs

Before creating bug reports, please check existing issues to avoid duplicates. When creating a bug report, include:

- A clear and descriptive title
- Steps to reproduce the problem
- Expected behavior vs actual behavior
- Screenshots if applicable
- Your environment (OS, browser, version)

### Suggesting Features

Feature suggestions are welcome! Please:

- Use a clear and descriptive title
- Provide a detailed description of the proposed feature
- Explain why this feature would be useful
- Include mockups or examples if possible

### Contributing Content

You can contribute:

- Past paper questions from exam boards
- Revision guides and study materials
- Flashcards and quiz questions
- Subject-specific resources

## Contributing Code

### Getting Started

1. Fork the repository
2. Clone your fork:
   ```bash
   git clone https://github.com/your-username/BrainMapRevision.git
   ```
3. Create a new branch:
   ```bash
   git checkout -b feature/your-feature-name
   ```
4. Make your changes
5. Test your changes thoroughly
6. Commit your changes (see commit guidelines below)
7. Push to your fork:
   ```bash
   git push origin feature/your-feature-name
   ```
8. Open a Pull Request

### Pull Request Process

1. Update the README.md with details of changes if applicable
2. Update documentation for any new features
3. Ensure all tests pass
4. Request review from maintainers
5. Address any feedback from reviewers
6. Once approved, your PR will be merged

### Code Standards

- Write clean, readable, and well-commented code
- Follow existing code style and conventions
- Include unit tests for new features
- Ensure your code is responsive and accessible
- Use meaningful variable and function names

## Contributing Past Paper Questions

We welcome contributions of past paper questions from official exam boards to help students practice effectively.

### Guidelines for Submitting Questions

1. **Verify authenticity**: Ensure the question is from an official exam board past paper (AQA, Edexcel, OCR, WJEC, SQA, etc.)

2. **Required information**:
   - Exam board name
   - Subject and level (e.g., GCSE, A-Level)
   - Year and session (e.g., June 2023)
   - Paper number and question number
   - Total marks available
   - The complete question text
   - Any associated diagrams or resources

3. **Include mark schemes**:
   - Official mark scheme
   - Examiner reports if available
   - Common mistakes to avoid

4. **Create a revision guide** (see below)

5. **Tag appropriately**:
   - Subject
   - Topic/subtopic
   - Difficulty level (Foundation/Higher or Easy/Medium/Hard)
   - Question type (multiple choice, short answer, essay, calculation, etc.)

6. **Fair use compliance**:
   - Questions must be used for educational purposes only
   - Properly attribute the source
   - Do not upload entire papers; focus on individual questions
   - Follow copyright and fair dealing guidelines

### Submission Format

Use the following template when submitting questions:

```markdown
## Question Details
- **Exam Board**: AQA
- **Subject**: Mathematics
- **Level**: GCSE Higher
- **Year**: 2023
- **Paper**: Paper 2
- **Question Number**: 12
- **Marks**: 4
- **Topics**: Algebra, Quadratic Equations

## Question
[Full question text here]

## Mark Scheme
[Official marking criteria]

## Examiner Comments
[Common mistakes, expected answers]

## Revision Guide Link
[Link to associated revision guide]
```

## Contributing Revision Guides

Revision guides help students understand the concepts behind past paper questions and learn how to approach similar problems.

### What Makes a Good Revision Guide?

1. **Clear explanations**: Break down complex concepts into understandable parts
2. **Step-by-step solutions**: Show the thought process, not just the answer
3. **Key concepts**: Highlight essential formulas, definitions, or principles
4. **Common mistakes**: Point out typical errors students make
5. **Practice tips**: Suggest strategies for tackling similar questions
6. **Additional examples**: Provide 2-3 similar worked examples
7. **Links to resources**: Reference relevant study materials or videos

### Revision Guide Template

```markdown
# [Topic Name] - Revision Guide

## Key Concepts

[List the fundamental concepts needed]

## Step-by-Step Approach

1. [First step with explanation]
2. [Second step with explanation]
3. [Continue...]

## Worked Example

[Show a complete worked solution with annotations]

## Common Mistakes

- [Mistake 1 and how to avoid it]
- [Mistake 2 and how to avoid it]

## Exam Tips

- [Tip 1]
- [Tip 2]

## Practice Questions

[Link to related past paper questions]

## Further Reading

[Links to additional resources]
```

### Quality Standards

- Use clear, correct English with proper grammar
- Ensure mathematical notation is properly formatted
- Include diagrams where helpful
- Verify all information is accurate
- Proofread before submitting

## Style Guidelines

### Markdown Formatting

- Use headers appropriately (H1 for titles, H2 for sections, etc.)
- Use code blocks for code examples
- Use bullet points for lists
- Use bold for emphasis sparingly
- Include alt text for images

### Writing Style

- Be clear and concise
- Use active voice
- Write for your audience (students aged 14-18 primarily)
- Avoid jargon unless necessary (and explain it when used)
- Be encouraging and supportive in tone

## Commit Message Guidelines

Write clear, descriptive commit messages:

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests when relevant

Examples:
```
Add quadratic equations revision guide
Fix navigation bug on mobile devices
Update contributing guidelines for past papers
```
